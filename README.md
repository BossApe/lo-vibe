# Musuhi

## 概要

Musuhi は、バイブコーディングをローカル環境で実行するためのツールです。本リポジトリでは Musuhi 自体の開発を行います。
- バイブコーディングとは<br>
AI を活用してソフトウェア開発のプロセスを効率化する手法であり、プロジェクト管理、要件定義、設計、コーディング、テストなどの各フェーズで AI を活用します。
- Musuhi の名前について<br>
「産霊（むすひ）」とは、神道においてモノを創造する力を指す言葉です。

## 開発方針

- 基本的に作業は AI に指示して実行し、AI を開発者として扱う
- 本プロジェクトの目的は、バイブコーディングをローカル環境で実践するためのツールを開発することである。そのため、バイブコーディングのプロセスを実践しながら開発を進める
- 開発手法は基本的にスクラム / MVP (Minimum Viable Product) に従う。ただし AI 活用を前提とするため、通常のデイリーミーティングは行わず、いくつかのタスク完了ごとに進捗と課題を振り返る時間を設ける
- タスク管理には GitHub Projects を使用する。更新は API 経由で AI に実施させる
- ソースバージョン管理には Jujutsu を使用する。Jujutsu を使った GitFlow を設計し、運用ルールをドキュメント化する
- 基本的にドキュメントは Markdown、図は Mermaid で記載し、GitHub で管理する
  - ドキュメントはプロジェクトの全フェーズで重要な役割を果たすため、適切に管理することが重要である
  - ドキュメントはプロジェクトの全フェーズで更新されるため、ドキュメントの管理方法を明確にすることが重要である
- コーディング規約を定め、コードの品質を保つことが重要である

## 開発の進め方

- プロンプトログは Markdown にまとめ、毎日ローカルに保存する
- AI 主体で開発を進めるため、通常のデイリーミーティングは行わず、いくつかのタスク完了ごとにプロジェクトの進捗や課題を振り返る時間を設ける
- 開発の各フェーズで必要なドキュメントを作成する
  - 要求仕様フェーズ: 要求仕様書
  - 要件定義フェーズ: プロジェクト計画書、要件定義書、品質管理、イテレーションロードマップ etc.
  - 初期設計・開発フェーズ: システム構成図、設計ドキュメント、操作仕様書 etc.
  - 拡張設計・開発フェーズ: 改修仕様書 etc.
- プロジェクトを進める前に、初期設計・開発フェーズまでのイテレーションロードマップを作成する
  - イテレーションロードマップ: イテレーションロードマップには、各イテレーションの目標、達成基準、タスクなどを記載する
- 要求仕様フェーズでは要求仕様書を作成する。要求仕様書には、プロジェクトの目的、背景、要求事項、制約条件など、Musuhi に必要な最小限の内容を記載する
- 要件定義フェーズでは、プロジェクト計画書 / 品質管理 / 要件定義書を作成する
  - プロジェクト計画書: プロジェクトの管理方法、タスク管理方法、ソースバージョン管理方法、ドキュメント管理方法などを記載する
  - 品質管理: コーディング規約、ドキュメント規約などを記載する
  - 要件定義書: システムの機能要件、非機能要件、ユーザーストーリーなどを記載する
- 初期設計・開発フェーズでは、機能を一つずつ、設計、コーディング、単体テスト、結合テスト、リリースのサイクルで進める
  - 設計: システム構成図、設計ドキュメント、操作仕様書などを作成する
  - コーディング: コーディング規約に従ってコードを書く
  - 単体テスト: 単体テストコードを書く
  - 結合テスト: 結合テストコードを書く
  - リリース: GitHub にリリースする
- イテレーションの区切りごとに、プロジェクトの進捗や課題を振り返る時間を設ける。振り返りではプロンプトログを活用し、AI との対話から得られた知見を分析する。得られた知見は次のイテレーション計画に活用し、必要に応じてプロジェクト計画書、要件定義書、設計ドキュメントへ反映する。また、拡張設計・開発フェーズに向けた拡張案・改修案を覚書として整理する。

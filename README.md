# Musuhi

## 概要

Musuhi は、バイブコーディングをローカル環境で実行するためのツールです。本リポジトリでは Musuhi 自体の開発をCopilotと共に行いますが、MusuhiのVersion 1.0が完了してからはMusuhiを活用してMusuhi自身の開発を行う予定です。

- バイブコーディングとは<br>
AI を活用してソフトウェア開発のプロセスを効率化する手法であり、プロジェクト管理、要件定義、設計、コーディング、テストなどの各フェーズで AI を活用します。
- Musuhi の名前について<br>
「産霊（むすひ）」とは、神道においてモノを創造する力を指す言葉です。

## 起動・停止手順（Docker Compose）

前提
- Docker Desktop（または Docker Engine）が起動していること

起動手順
1. Musuhi のルートディレクトリに移動
   ```bash
   cd <Musuhiルートディレクトリパス>
   ```
2. Docker Compose をバックグラウンドで起動
   ```bash
   docker compose -f _compose/docker-compose.yml up -d
   ```
3. 起動確認
   ```bash
   docker compose -f _compose/docker-compose.yml ps
   ```

停止手順
1. サービスを停止
   ```bash
   docker compose -f _compose/docker-compose.yml down
   ```

補足
- ログ確認
  ```bash
  docker compose -f _compose/docker-compose.yml logs -f
  ```

## メモ（不要になったら削除）
下記はMusuhiでの「4.1 UC-01 新規プロジェクト開発」機能を詳細に分解した業務フロー案です。下記の業務フローと矛盾しているところがないか「musuhi/_document/002.要件定義フェーズ」、及び、「musuhi/_document/003.設計・開発・テストフェーズ」ディレクトリ配下をリファクタリングしてください

Step1.
ユーザが作りたいシステムの概要を UI で箇条書き・メモ書き形式でインプットする

Step2.
Musuhiがインプットをもとに必要な機能・構成要素を洗い出し、アプリケーション名（=プロジェクト名）を生成し、Musuhiシステムのあるディレクトリと同じ階層にプロジェクト名（以下、新規プロジェクト）のディレクトリとその配下にディレクトリ構成テンプレートから必須なディレクトリを生成する。空白なディレクトリには.keepファイルを配置しておくこと

Step3.
MusuhiがGitHub（別のGitHub相当のツールを選択できるようにするのは本フェーズ対象外）に新規プロジェクトを作成する。プロジェクト作成に必要で不明な項目はユーザに問い合わせる。新規プロジェクト全体をcommit/pushする

Step4.
Musuhiが新規プロジェクト配下にGitHub Projectsを作成する。Phase0の各種タスクを生成

Step5.
MusuhiがTK0-1-1を進める。適宜タスクを更新する。
自動生成し、リファクタリングを繰り返し、修正項目がなくなればタスク完了。以降、タスク完了毎に「新規プロジェクト/_document/000.進捗状況」ディレクトリにgenerate_issue_roadmapを使用して進捗ファイルを出力し、生成したドキュメント全てをcommit / pushする。

Step6.
MusuhiとユーザでTK0-1-2を進める。適宜タスクを更新する。
リファクタリングとユーザレビューを繰り返し、修正項目・ユーザ指摘項目がなくなればタスク完了

Step7.
MusuhiがTK0-2-1を進める。適宜タスクを更新する。
自動生成し、リファクタリングを繰り返し、修正項目がなくなればタスク完了。

Step8.
MusuhiとユーザでTK0-2-2を進める。適宜タスクを更新する。
リファクタリングとユーザレビューを繰り返し、修正項目・ユーザ指摘項目がなくなればタスク完了

Step9.
MusuhiがTK0-3-1を進める。適宜タスクを更新する。
自動生成し、リファクタリングを繰り返し、修正項目がなくなればタスク完了。

Step10.
MusuhiがTK0-4-1を進める。適宜タスクを更新する。
自動生成し、リファクタリングを繰り返し、修正項目がなくなればタスク完了。

Step11.
MusuhiがTK0-4-2を進める。適宜タスクを更新する。
新規プロジェクトで必要な「Musuhi/tools」配下のツールを「新規プロジェクト/tools」配下にコピーする。

Step12.
MusuhiがTK1-1-1から1機能の受入テスト・masterブランチコミットまで実施する。適宜タスクを更新する。
各Ticketの機能開発は設計・リファクタリング・開発・リファクタリング・単体テスト・リファクタリング・結合テスト・リファクタリングを実施する。リファクタリングは修正項目がなくなるまで繰り返す
設計では「musuhi/_document/004.リリース・運用フェーズ」配下にチケットタスクの機能が含まれる業務フロー図（要件定義で作成されたユースケースフロー図を詳細化）や他、必要な設計書を編集してください

Step13.
ユーザ（参加任意）とMusuhiがTK1-1-y: レトロスペクティブを実施する。
改善・追加機能がある場合はレトロスペクティブの結果と合わせて「musuhi/_document/003.設計・開発・テストフェーズ」配下の適当なディレクトリに保存する
次SprintのTicket分割を実施し、タスクを登録する。次以降のSprintに修正がある場合はその修正も行う

Step14.
下記の必要がある場合は各SprintタスクにTicketタスクを追加し、順次実施する
PH2: リリース・運用
  SP2-1: リリースIaC生成
  SP2-2: 新規システム起動・停止手順ドキュメント生成
  SP2-3: エンドユーザ向けドキュメント生成

# Lo-Vibe

## 概要

Lo-Vibe は、Vibecodingをローカル環境で実施するためのツールです。本プロジェクトはLo-Vibeの開発を行うプロジェクトです。
Vibecodingとは、AIを活用してソフトウェア開発のプロセスを効率化する手法であり、プロジェクト管理、要件定義、設計、コーディング、テストなどの各フェーズで AI を活用します。

## 開発方針

- 基本的に作業はAIに指示して実行する。AIは開発者として扱う
- 本プロジェクトの目的は、Vibecodingをローカル環境で実施するためのツールを開発することであるため、Vibecodingのプロセスを実践することが重要である
- 開発手法は基本的にスクラム / MVP (Minimum Viable Product) の手法に従うが、AIを活用するため通常のスクラムの開発のデイリーミーティングは行わず、いくつかのタスクの完了毎にプロジェクトの進捗や課題を振り返る時間を設ける
- タスク管理を GitHub Projects を使って行う。記載は API でAIに実施させるようにする
- ソースバージョン管理は Jujutsu を使用する。Jujutsu を使った GitFlow を考え、運用ルールをドキュメント化する
- 基本的にドキュメントは Markdown、図は Mermaid で記載し、ドキュメントの管理は GitHub で行う
  - ドキュメントはプロジェクトの全フェーズで重要な役割を果たすため、適切に管理することが重要である
  - ドキュメントはプロジェクトの全フェーズで更新されるため、ドキュメントの管理方法を明確にすることが重要である
- コーディング規約を定め、コードの品質を保つことが重要である

## 開発の進め方

- プロンプトログは Markdownにまとめ毎日ローカルに保存する
- AIで開発を進めるため通常のスクラムの開発のデイリーミーティングは行わず、いくつかのタスクの完了毎にプロジェクトの進捗や課題を振り返る時間を設ける
- 開発の各フェーズで必要なドキュメントを作成する
  - 要求仕様フェーズ: 要求仕様書
  - 要件定義フェーズ: プロジェクト計画書、要件定義書、品質管理、イテレーションロードマップ etc.
  - 初期設計・開発フェーズ: システム構成図、設計ドキュメント、操作仕様書 etc.
  - 拡張設計・開発フェーズ: 改造仕様書 etc.
- プロジェクトを進める前に、初期設計・開発フェーズまでのイテレーションロードマップを作成する
 - イテレーションロードマップ: イテレーションロードマップには、各イテレーションの目標、達成基準、タスクなどを記載する
- 要求仕様フェーズでは、要求仕様書を作成する。要求仕様書には、プロジェクトの目的、背景、要求事項、制約条件などをLo-vibeに必要な最小限の内容として記載する
- 要件定義フェーズでは、プロジェクト計画書 / 品質管理 / 要件定義書を作成する。
 - プロジェクト計画書: プロジェクトの管理方法、タスク管理方法、ソースバージョン管理方法、ドキュメント管理方法などを記載する。
 - 品質管理: コーディング規約、ドキュメント規約などを記載する
 - 要件定義書: システムの機能要件、非機能要件、ユーザーストーリーなどを記載する
- 初期設計・開発フェーズでは、機能一つづつ、設計、コーディング、単体テスト、結合テスト、リリースのサイクルで進める
  - 設計: システム構成図、設計ドキュメント、操作仕様書などを作成する
  - コーディング: コーディング規約に従ってコードを書く
  - 単体テスト: 単体テストコードを書く
  - 結合テスト: 結合テストコードを書く
  - リリース: GitHub にリリースする
- イテレーションの区切りごとに、プロジェクトの進捗や課題を振り返る時間を設ける。その際、プロンプトログを振り返り、プロジェクトの進捗や課題を分析する。次のイテレーションの計画を立てる際に、プロンプトログから得られた知見を活用する。振り返りで得られた知見は、プロジェクトの計画書や要件定義書、設計ドキュメントなどに反映する。プロジェクトの進捗や課題を振り返る際には、プロンプトログを活用して、AIとの対話から得られた知見を分析することが重要である。また、振り返りの際得られた知見を元に拡張設計・開発フェーズでの拡張案・改造案を覚書としてまとめる。

### プロジェクト全体のディレクトリ構成概要

```text
lo-vibe/
├── README.md             # プロジェクト全体の概要と開発方針を記載
├── _document/            # ドキュメント全般を格納
│   ├── plan/              # プロジェクト計画関連ドキュメント
│   ├── requirement/       # 要求仕様関連ドキュメント
│   ├── definition/        # 要件定義関連ドキュメント
│   └── design/            # 設計関連ドキュメント
├── _compose/             # プロジェクトのシステム構成をコード化するディレクトリ
├── <サービス名>/           # ソースコード全般を格納
│   └── <サービス関連のリソース>  # サービス関連のリソース（例: API 定義、DB マイグレーションファイルなど）
├── test/                 # テストコード全般を格納
│   ├── unit/             # 単体テストコード
│   └── integration/      # 結合テストコード
└── tools/                # 開発・運用を効率化するツール類を格納
    ├── README.md         # ツールディレクトリの概要とツール一覧を記載
    └── save_promptinglog/ # プロンプトログ保存ツール (Go)
```


- ドキュメント
  - 要求仕様フェーズ
  - 要件定義フェーズ
    - プロジェクト計画
      - プロジェクト管理方法: Scrum
      - タスク管理: GitHub Projects をカンバン形式で管理
      - ソースバージョン管理: JuJutsu
      - JuJutsu を使った GitFlow を考える
    - 要件定義書
  - 設計フェーズ
    - 規約
      - コーディング規約
      - ドキュメント規約
    - システム構成
    - イテレーションロードマップ
- ソースコード
  - コーディング規約に従う
  - 単体テストコード
  - 結合テストコード
- ツール
  - プロンプトログ保存ツール (Go)
  - その他、開発・運用を効率化するツール類

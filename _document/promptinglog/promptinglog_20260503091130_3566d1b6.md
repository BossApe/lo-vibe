要件を把握しました。GitHub Projectsにはまだ登録せず、設計・開発・テストフェーズ向けに「フェーズタスク→イテレーションタスク→チケットタスク」の3階層で、ガントチャート的に進捗確認しやすい分割案を具体的に整理して提示します。以下の進め方だと、GitHub Projectsでガントチャート風に見やすく、運用もしやすいです。  
まだ登録せず、まずは設計案として使える形でまとめます。

**全体方針**
1. 階層は3段に固定する  
2. 上位ほど成果物ベース、下位ほど実装作業ベースにする  
3. 期間を持たせるのは「フェーズ」と「イテレーション」を中心にする  
4. チケットは1〜2日で終わる粒度にする  
5. 依存関係を明示して、Roadmapビューで時系列確認する

**タスク階層（推奨）**
- フェーズタスク: Epic相当  
  例: 設計フェーズ、開発フェーズ、テストフェーズ
- イテレーションタスク: Feature相当  
  例: 設計-Iteration1 基盤設計、開発-Iteration2 API/BFF実装
- チケットタスク: Issue相当  
  例: mmmm-rdbのDDL作成、bookman-bffのルーティング実装、E2Eケース作成

**期間の目安（例）**
- 設計フェーズ: 2週間  
- 開発フェーズ: 5週間  
- テストフェーズ: 3週間  
- 合計: 10週間

**WBS案（そのままProjectsに落とせる粒度）**

**GitHub Projectsの項目設計（重要）**
- `Type`: Phase / Iteration / Ticket
- `Status`: Backlog / Ready / In Progress / Blocked / Review / Done
- `Phase`: 設計 / 開発 / テスト
- `Iteration`: I1-1, I1-2, I2-1...
- `Start date`
- `Target date`
- `Service`: apigw / bff / bl / rdb / ui / web / cross
- `Priority`: P0/P1/P2
- `Estimate`: XS/S/M/L（または Story Point）
- `Parent`: 親タスクID（手動リンク）
- `Depends on`: 依存先Issue

**ガントチャート風に見るためのビュー構成**
1. Roadmapビュー  
- Group by: `Phase`  
- Timeline: `Start date`〜`Target date`  
- Filter: `Type:Phase,Iteration`中心（必要時Ticketも表示）

2. Boardビュー  
- Group by: `Status`  
- Filter: `Type:Ticket`  
- 日々の運用はここが中心

3. Tableビュー  
- 依存関係、見積、担当、遅延確認に使用  
- Sort: `Target date` → `Priority`

**分割ルール（チケット化の判断基準）**
- 1チケットは「成果物が1つ」「担当が1人」「完了条件が明確」
- 2日を超える作業は分割する
- 複数サービスに跨る作業はサービス別に分ける
- テストは実装チケットと分離し、追跡可能にする
- 不具合修正は元チケットとリンクする（再発分析しやすくする）

**命名規約（見やすさ重視）**
- フェーズ: `PH1 設計フェーズ`
- イテレーション: `IT2-2 Backend実装`
- チケット: `[BL] 書籍検索API実装` のように先頭タグでサービス識別
- 例: `[RDB] 初期DDL作成`, `[UI] 書籍一覧画面実装`

この案であれば、上位は計画管理、下位は実行管理に役割分担でき、進捗遅延の箇所もすぐ特定できます。

## GitHub Projectsの2種類

| 種類 | URL | 作成方法 |
|------|-----|---------|
| **ユーザーレベル**（今回） | `github.com/users/BossApe/projects/1` | `--owner BossApe` |

## どちらが適切か

**リポジトリレベル**が適切なケース（Musuhiの場合はこちら）:
- 単一リポジトリのIssue/PRと連携したい
- リポジトリのCollaboratorがProjectにアクセスできるようにしたい
- リポジトリの「Projects」タブから見える状態にしたい

**ユーザー/組織レベル**が適切なケース:
- 複数リポジトリをまたいで管理したい
- 組織全体のロードマップを管理したい

**① GitHub UI上でMusuhiリポジトリをProjectに紐付ける（推奨・簡単）**
- Project設定（⚙️）→ `Linked repositories` → `BossApe/Musuhi` を追加
- これにより既存ProjectにMusuhiのIssueを紐付け可能になります

**② 作り直す（完全移行）**
```bash

```

#### ユーザー (2026-05-02 23:25:34 JST)

draft issue と GitHub Issueの違いは？

#### アシスタント (2026-05-02 23:25:43 JST)

要点だけ言うと、Draft issue はProject内だけの仮タスク、GitHub Issue はリポジトリに属する正式チケットです。

主な違いは次の通りです。

1. 所属先
- Draft issue: GitHub Projects内にだけ存在
- GitHub Issue: 特定リポジトリに存在

2. URLと識別
- Draft issue: リポジトリの issue 番号を持たない
- GitHub Issue: owner/repo#123 のような番号付きで参照可能

3. 開発連携
- Draft issue: ブランチ連携、PRの自動クローズ、sub-issue親子などのIssue機能が弱い/使えない
- GitHub Issue: Assignee/Label/Milestone、PR連携、sub-issue、検索など標準機能が使える

4. 用途
- Draft issue: アイデア出し、一次分解、未確定タスクの整理
- GitHub Issue: 実装・レビュー・完了まで追う本番タスク管理

5. 変換
- Draft issue は後から GitHub Issue に変換可能
- 変換後はリポジトリIssueとして運用できる

今回の親子ネスト表示の観点では、Draft issue + Parentテキストより、GitHub Issue + sub-issue の方が構造的に扱いやすいです。

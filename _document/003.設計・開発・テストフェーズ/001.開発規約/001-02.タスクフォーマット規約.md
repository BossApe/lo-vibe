## 1. 目的

本ドキュメントは、フェーズ・スプリント・チケットの記載フォーマットを統一し、
レビュー時に「何が完了したか」を判定しやすくすることを目的とする。

## 2. 階層ごとの記載ルール

| 項目 | フェーズタスク | スプリントタスク | チケットタスク |
| --- | --- | --- | --- |
| 期間 | 参考（開発Phase単位） | 参考（Sprint単位） | 設定しない（AIが実施） |
| 開始日時 | 任意 | 任意 | **必須**（タスク開始時に記入） |
| 終了日時 | 任意 | 任意 | **必須**（タスク完了時に記入） |
| AI Token数 | 任意 | 任意 | **必須**（タスク完了時に記入） |
| 成果物 | 必須（複数可） | 必須（対象FR・AC） | 必須（1機能スライス） |
| 完了条件 | 必須 | 必須（AC Pass・リファクタリングPass・masterマージ） | 必須（設計・実装・テスト・リファクタリングの各完了条件） |
| 親子関係 | 親なし | 親はフェーズ（Phase0/1/2） | 親はスプリント |
| 粒度 | Phase0/1/2 単位 | FR 単位または Sprint 単位 | 1機能スライス（単一責務・独立テスト可能） |

### タイトル形式

- フェーズ: `PHx: {フェーズ名}`
- スプリント: `ITx-y: {目的}`
- チケット: `TKx-y-z: [領域] {何をするか（端的に）}`

タイトルのルール:

- **端的に・一行で** 収まる長さにする
- FR コード番号・詳細説明はタイトルに含めず、本文の背景・実施内容に記載する
- 「実装」「自動生成」「登録」など **動詞で終わる** 形式を推奨する

タイトル例:

- `PH1: 設計フェーズ`
- `SP2-2: 開発フェーズ Backend機能実装`
- `TK1-1-1: [core] システム概要の入力・保存を実装`
- `TK0-2-1: [all] 要件定義書を自動生成`

## 3. Issueテンプレート

### 3.1 フェーズタスク

```markdown
## 目的

## 対象期間
- 開始:
- 完了:

## 配下スプリント
- ITx-y:

## 完了条件
- [ ] 対象スプリントがすべて完了
- [ ] フェーズ成果物レビューが完了
```

### 3.2 スプリントタスク

```markdown
## 目的

## スコープ

## 対象チケット
- #

## 完了条件
- [ ] 必須チケット完了率 100%
- [ ] テスト結果が記録されている
- [ ] 次スプリントへの引継ぎ事項が記録されている
```

### 3.3 チケットタスク

> **Sprint 内最初のチケットには、実施内容の先頭に以下を追加すること。**
>
> ```
> - **【Sprint開始】** 機能業務フロー図とインフラ構成図を作成する
>   - 本 Sprint で実装する機能の業務フロー図を `_document/003.設計・開発・テストフェーズ/` 配下に作成する
>   - インフラ構成図（サービス・DB・外部API連携の概要）を同ディレクトリに作成する
>   - commit / push する
> ```
>
> 作業工程チェックリストの先頭行・受け入れ条件にも同様に追加する。

**タスク計測フィールドの記入タイミング:**

| フィールド | 記入タイミング | 形式 |
| --- | --- | --- |
| 開始日時 | チケット作業開始直前（Issueを In Progress に移動する時点） | `yy-mm-dd hh:mm:ss` |
| 終了日時 | チケット作業完了直後（Done に移動する時点） | `yy-mm-dd hh:mm:ss` |
| AI Token数 | タスク完了時に会話ウィンドウ等から合計値を確認して記入 | 整数（例: `125,430`） |

```markdown
## 背景

## 実施内容
<!-- 箇条書きで詳細に記述する。手順・対象・設計方針・仕様を含める。 -->
<!-- 例: -->
<!-- - 〇〇 API（POST /xxx）を設計・実装する -->
<!--   - リクエスト: 〇〇・〇〇 -->
<!--   - バリデーション: 〇〇必須 -->

<!-- - 単体テスト・結合テストを実装し CI Green を確認する -->
```

## 4. Definition of Done

チケットを `Done` にする条件:

- 設計: 機能スライスの設計方針を記録済み（コメント・設計メモ等）
- 実装: 受け入れ条件の動作が確認できる状態
- テスト: 単体・結合・APIテストが自動テストとして実装済み・CI Green
- リファクタリング: 修正箇所が0になるまで繰り返し完了
- 成果物出力: タスク完了時に成果物（例: 進捗HTML、ログ、レポート）を `新規プロジェクト/_document/000.進捗状況` へ保存済み
- 出力記録: 出力先パスと生成ファイル名（時刻付き）をチケットコメントまたは関連ドキュメントへ記録済み
- 関連ドキュメント更新が必要な場合は反映済み
- 生成・更新されたドキュメントが commit / push 済み
- 回帰影響の確認結果を記録済み
- タスク計測: 開始日時・終了日時・AI Token数がチケットに記録済み

## 5. 見積基準

> 開発作業はAIが実施するため、日数による見積は行わない。
> 過去チケットの **所要時間・AI Token数の実績** を基準に Story Point（SP）で相対評価する。

### 5.1 Story Point 定義

| SP | 所要時間の目安 | AI Token数の目安 | 例 |
| --- | --- | --- | --- |
| 1 | ～30分 | ～50,000 | 誤記修正、軽微な設定値更新 |
| 2 | 30分〜1時間 | 50,000〜150,000 | 単一API実装（正常系のみ） |
| 3 | 1〜3時間 | 150,000〜400,000 | API + バリデーション + 単体テスト |
| 5 | 3〜8時間 | 400,000〜800,000 | 外部API連携・モック設計が必要 |
| 8 | 8時間超 | 800,000超 | 複数サービス横断・大規模リファクタリング |

> SPの目安はプロジェクト初期はこの表を参照する。Sprintの実績が蓄積された後は
> レトロスペクティブで実績値に基づいて補正する。

### 5.2 新チケット作成時のSP見積手順

1. 過去チケットの `タスク計測`（所要時間・AI Token数）を参照する
2. 類似チケットと比較して相対的にSPを決定する
3. 実績がない場合は5.1の目安表を初期値とする
4. 見積SPをGitHub Projectsの `Estimate` フィールドに設定する

### 5.3 ベロシティ計算

$$\text{Sprint ベロシティ} = \sum_{\text{Done チケット}} \text{SP}$$

- レトロスペクティブ時に前Sprintのベロシティを記録する
- 次Sprintのスコープ設定の参考にする

## 6. 申し送り事項・改善要望の記録ルール

### 6.1 記録タイミング

リファクタリング（設計・実装・単体テスト・結合テストの各工程後）を実施している際、
**現在のチケットスコープ外**と判断される機能追加・改善点が判明した場合、
**その場でチケットにコメントを追加し、申し送り事項として記録する。**

### 6.2 記録フォーマット

チケットの `## 申し送り事項・改善要望` セクション（またはGitHub Issue コメント）に以下の形式で記録する。

```markdown
### 申し送り [発見日時: yy-mm-dd]

- **発見工程**: 実装リファクタリング / 設計リファクタリング / 単体テストリファクタリング / 結合テストリファクタリング
- **内容**: （何を改善すべきか・何を別タスク化すべきか）
- **理由**: （なぜ現チケットで対応しないか）
- **優先度**: 高 / 中 / 低
- **チケット化**: レトロスペクティブ時に検討 / 次Sprint TK予定
```

### 6.3 レトロスペクティブでの取り込み手順

1. Sprint内の全チケットの `申し送り事項・改善要望` セクションを収集する
2. 重複・関連する申し送りを整理・統合する
3. 優先度・依存関係を評価し、次以降のSprintのチケットとしてGitHub Projectsに登録する
4. 登録したチケット番号を元の申し送り記録に追記する（`→ #xxx として登録済み`）

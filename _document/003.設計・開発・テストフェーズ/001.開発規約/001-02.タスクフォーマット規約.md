## 1. 目的

本ドキュメントは、フェーズ・イテレーション・チケットの記載フォーマットを統一し、
レビュー時に「何が完了したか」を判定しやすくすることを目的とする。

## 2. 階層ごとの記載ルール

| 項目 | フェーズタスク | イテレーションタスク | チケットタスク |
| --- | --- | --- | --- |
| 期間 | 参考（開発Phase単位） | 参考（Iteration単位） | 設定しない（AIが実施） |
| 成果物 | 必須（複数可） | 必須（対象FR・AC） | 必須（1機能スライス） |
| 完了条件 | 必須 | 必須（AC Pass・リファクタリングPass・masterマージ） | 必須（設計・実装・テスト・リファクタリングの各完了条件） |
| 親子関係 | 親なし | 親はフェーズ（Phase0/1/2） | 親はイテレーション |
| 粒度 | Phase0/1/2 単位 | FR 単位または Iteration 単位 | 1機能スライス（単一責務・独立テスト可能） |

### タイトル形式

- フェーズ: `PHx: {フェーズ名}`
- イテレーション: `ITx-y: {目的}`
- チケット: `TKx-y-z: [領域] {何をするか（端的に）}`

タイトルのルール:

- **端的に・一行で** 収まる長さにする
- FR コード番号・詳細説明はタイトルに含めず、本文の背景・実施内容に記載する
- 「実装」「自動生成」「登録」など **動詞で終わる** 形式を推奨する

タイトル例:

- `PH1: 設計フェーズ`
- `IT2-2: 開発フェーズ Backend機能実装`
- `TK1-1-1: [core] システム概要の入力・保存を実装`
- `TK0-2-1: [all] 要件定義書を自動生成`

## 3. Issueテンプレート

### 3.1 フェーズタスク

```markdown
## 目的

## 対象期間
- 開始:
- 完了:

## 配下イテレーション
- ITx-y:

## 完了条件
- [ ] 対象イテレーションがすべて完了
- [ ] フェーズ成果物レビューが完了
```

### 3.2 イテレーションタスク

```markdown
## 目的

## スコープ

## 対象チケット
- #

## 完了条件
- [ ] 必須チケット完了率 100%
- [ ] テスト結果が記録されている
- [ ] 次イテレーションへの引継ぎ事項が記録されている
```

### 3.3 チケットタスク

> **Iteration 内最初のチケットには、実施内容の先頭に以下を追加すること。**
>
> ```
> - **【Iteration開始】** 機能業務フロー図とインフラ構成図を作成する
>   - 本 Iteration で実装する機能の業務フロー図を `_document/003.設計・開発・テストフェーズ/` 配下に作成する
>   - インフラ構成図（サービス・DB・外部API連携の概要）を同ディレクトリに作成する
>   - commit / push する
> ```
>
> 作業工程チェックリストの先頭行・受け入れ条件にも同様に追加する。

```markdown
## 背景

## 実施内容
<!-- 箇条書きで詳細に記述する。手順・対象・設計方針・仕様を含める。 -->
<!-- 例: -->
<!-- - 〇〇 API（POST /xxx）を設計・実装する -->
<!--   - リクエスト: 〇〇・〇〇 -->
<!--   - バリデーション: 〇〇必須 -->
<!-- - DB モデル（xxx テーブル）を設計しマイグレーションファイルを作成する -->
<!-- - 単体テスト・結合テストを実装し CI Green を確認する -->

## 作業工程チェックリスト
- [ ] 設計: 機能スライスの設計方針・API I/F・バリデーションルールを確認
- [ ] 実装: 機能を実装
- [ ] テスト: 単体・結合・APIテストを実装し CI Green を確認
- [ ] リファクタリング: 修正箇所が0になるまで繰り返す

## 受け入れ条件
- [ ] 条件1
- [ ] 条件2

## 依存関係
- Depends on: #

## 影響範囲
- 対象: 
- 非対象:
```

## 4. Definition of Done

チケットを `Done` にする条件:

- 設計: 機能スライスの設計方針を記録済み（コメント・設計メモ等）
- 実装: 受け入れ条件の動作が確認できる状態
- テスト: 単体・結合・APIテストが自動テストとして実装済み・CI Green
- リファクタリング: 修正箇所が0になるまで繰り返し完了
- 成果物出力: タスク完了時に成果物（例: 進捗HTML、ログ、レポート）を `新規プロジェクト/_document/000.進捗状況` へ保存済み
- 出力記録: 出力先パスと生成ファイル名（時刻付き）をチケットコメントまたは関連ドキュメントへ記録済み
- 関連ドキュメント更新が必要な場合は反映済み
- 生成・更新されたドキュメントが commit / push 済み
- 回帰影響の確認結果を記録済み

## 5. 見積基準

> 開発作業はAIが実施するため、日数による見積は行わない。
> 複雑さ（変更範囲・依存関係・テスト難度）で相対評価する。

| 見積 | 目安 | 例 |
| --- | --- | --- |
| XS | 変更が1モジュール以内・テスト直書き可能 | 誤記修正、軽微な設定値更新 |
| S | 変更が1〜2モジュール・テストパターンが少ない | 単一API実装（正常系のみ） |
| M | 変更が複数モジュール・外部依存なし | API + バリデーション + 単体テスト |
| L | 外部API連携・モック設計が必要 | 外部ツール連携API・非同期処理 |

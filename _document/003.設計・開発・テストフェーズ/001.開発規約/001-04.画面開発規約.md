## 1. 目的

本ドキュメントは、Musuhi フロントエンド（SvelteKit）における画面開発の実装規約を定める。
コードの一貫性・保守性・品質を担保し、チームメンバー間での認識齟齬を防ぐことを目的とする。

---

## 2. 適用範囲

- 対象ディレクトリ: `services/musuhi-frontend/`
- 対象成果物: SvelteKit ページコンポーネント（`.svelte`）・型定義（`.ts`）・スタイル
- 対象者: フロントエンド開発に携わる全作業者

---

## 3. 技術スタック

| カテゴリ | 技術・バージョン |
| --- | --- |
| フレームワーク | SvelteKit（Svelte 5） |
| 言語 | TypeScript |
| ビルドツール | Vite 8 |
| スタイル | Scoped CSS（`<style>` ブロック内） |
| パッケージマネージャ | npm |
| 型チェック | `svelte-check` + `tsc` |

## 4. ディレクトリ・ファイル構成

```text
services/musuhi-frontend/src/
├── app.d.ts                    # グローバル型定義（window 拡張等）
├── app.html                    # エントリポイント HTML
├── lib/
│   ├── index.ts                # lib エクスポート
│   ├── assets/                 # 画像・アイコン等の静的アセット
│   └── components/             # 再利用可能コンポーネント（将来追加時）
└── routes/
    ├── +layout.svelte          # グローバルレイアウト
    └── {feature}/
        └── +page.svelte        # 各画面コンポーネント
```

### 配置ルール

| 種別 | 配置先 |
| --- | --- |
| ページ固有コンポーネント | `routes/{feature}/+page.svelte` に直書き（小規模時） |
| 複数ページで再利用するコンポーネント | `lib/components/{ComponentName}.svelte` |
| グローバル型定義 | `app.d.ts` |
| フィーチャー固有の型定義 | `lib/types/{feature}.ts` |
| 静的アセット | `lib/assets/` または `static/` |

## 5. コンポーネント設計規約

### 5.1 ページコンポーネント

各画面は SvelteKit のファイルベースルーティングに従い `routes/{feature}/+page.svelte` として実装する。

**スクリプトブロック構成順序（厳守）:**

```svelte
<script lang="ts">
  // 1. import 文
  import { goto } from '$app/navigation';

  // 2. 環境変数・定数
  const API_BASE = import.meta.env.VITE_API_BASE_URL ?? '';
  const MAX_LENGTH = 4096;

  // 3. URLパラメータ受け取り（必要な場合）
  const { url } = $props();
  const overviewId = url?.searchParams.get('overviewId') ?? '';

  // 4. 状態変数（$state）
  let content = $state('');
  let isSubmitting = $state(false);
  let errorMessage = $state('');

  // 5. 派生値・$derived（必要な場合）
  let charCount = $derived(content.length);

  // 6. バリデーション関数
  function validate(): boolean { ... }

  // 7. イベントハンドラ
  async function handleSubmit(e: Event): Promise<void> { ... }
</script>
```

### 5.2 状態管理

- ページ内ローカル状態は Svelte 5 の `$state` / `$derived` ルーン を使用する
- 複数ページにまたがる状態は `$lib/stores/` に Svelte store を定義する（将来対応）
- `writable` / `readable` / `derived` の使い分けは別途ストア設計規約で定める

**状態変数の命名規則:**

| 種別 | 命名パターン | 例 |
| --- | --- | --- |
| 入力値 | `{fieldName}` (camelCase) | `content`, `projectName` |
| ローディング | `is{Action}ing` | `isSubmitting`, `isLoading` |
| エラーメッセージ | `{scope}ErrorMessage` または `errorMessage` | `errorMessage` |
| 成功メッセージ | `{scope}SuccessMessage` または `successMessage` | `successMessage` |
| リスト | `{noun}List` | `featureList` |

### 5.3 API 通信

**必須ルール:**

1. ベース URL は環境変数 `VITE_API_BASE_URL` から取得し、定数 `API_BASE` に格納する
2. `fetch` の呼び出しは必ず `try / catch / finally` で囲む
3. `finally` ブロックで `isSubmitting = false` を確実に戻す
4. POST / PUT / PATCH のリクエストは必ず `Content-Type: application/json` ヘッダーを付与する
5. リクエスト送信前に `errorMessage = ''` / `successMessage = ''` をリセットする

**HTTP メソッドの使い分け:**

| 操作 | メソッド | 成功ステータス |
| --- | --- | --- |
| 新規作成 | POST | 201 Created |
| 全体更新（既存レコード上書き） | PUT | 200 OK |
| 部分更新 | PATCH | 200 OK |
| 取得 | GET | 200 OK |
| 削除 | DELETE | 204 No Content |

**レスポンス処理パターン（標準）:**

```typescript
async function handleSubmit(e: Event): Promise<void> {
  e.preventDefault();
  errorMessage = '';
  if (!validate()) return;

  isSubmitting = true;
  try {
    const res = await fetch(`${API_BASE}/api/v1/...`, {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ ... })
    });

    if (res.status === 201) {
      const data = await res.json();
      await goto(`/next-page?id=${data.data.id}`);
    } else if (res.status === 422) {
      const data = await res.json();
      errorMessage = data.error?.message ?? 'バリデーションエラーが発生しました。';
    } else {
      errorMessage = 'サーバーエラーが発生しました。しばらく待ってから再試行してください。';
    }
  } catch {
    errorMessage = 'ネットワークエラーが発生しました。接続を確認してください。';
  } finally {
    isSubmitting = false;
  }
}
```

### 5.4 エラーハンドリング

| エラー種別 | 処理方法 | 表示箇所 |
| --- | --- | --- |
| フロントバリデーション失敗 | `errorMessage` にセットして送信中止 | フォーム上部または入力欄近傍 |
| API 422 | レスポンスの `error.message` を `errorMessage` に表示 | フォーム上部 |
| API 4xx / 5xx（422以外） | 固定メッセージを `errorMessage` に表示 | フォーム上部 |
| ネットワーク断（fetch 例外） | 固定メッセージを `errorMessage` に表示 | フォーム上部 |

- `errorMessage` が空でない場合、ユーザが視認できる位置に赤字等で表示する
- `successMessage` が空でない場合、グリーン系でフィードバックを表示する

## 6. スタイル規約

- スタイルは原則 `<style>` ブロックに Scoped CSS で記述する（グローバル汚染禁止）
- グローバルスタイルが必要な場合は `app.html` または `+layout.svelte` の `:global()` セレクタを使用する
- クラス名は kebab-case（`form-container`, `error-message` 等）を使用する
- ユーティリティクラスを多数定義する場合は `lib/styles/` に切り出す（将来対応）
- カラー・サイズ等の定数は CSS カスタムプロパティ（`--color-primary` 等）で管理する

**禁止事項:**

- `<style>` ブロック外へのインラインスタイル記述（`style="..."` 属性）は原則禁止とする
  - ただし動的スタイルバインディング（`:style`）は許容する

## 7. 命名規約

| 対象 | 規則 | 例 |
| --- | --- | --- |
| ファイル名（ページ） | SvelteKit の規約に従う | `+page.svelte`, `+layout.svelte` |
| ファイル名（コンポーネント） | PascalCase | `OverviewForm.svelte` |
| ファイル名（TypeScript） | camelCase | `systemOverview.ts` |
| 変数・関数 | camelCase | `handleSubmit`, `overviewId` |
| 型・インターフェース | PascalCase | `SystemOverview`, `ApiResponse` |
| 定数（不変値） | UPPER_SNAKE_CASE または `const` camelCase | `API_BASE`, `MAX_LENGTH` |
| CSS クラス | kebab-case | `form-container`, `submit-btn` |

## 8. 型定義規約

- `any` 型の使用は禁止する。不明な型は `unknown` を使用してから型ガードで絞り込む
- API レスポンス型は `lib/types/` に定義し、`fetch` 結果をキャストして使用する
- Svelte の `$props()` を使うコンポーネントは Props 型を明示する

**API レスポンス型定義の標準パターン:**

```typescript
// lib/types/systemOverview.ts
export interface SystemOverview {
  id: string;
  content: string;
  createdAt: string;
}

export interface ApiResponse<T> {
  data: T;
}

export interface ApiError {
  error: {
    code: string;
    message: string;
    details?: { field: string; message: string }[];
  };
}
```

## 9. アクセシビリティ規約

- フォームの `<input>` / `<textarea>` には必ず `id` と対応する `<label for="...">` を付与する
- 送信ボタンは `disabled={isSubmitting}` で多重送信を防止する
- エラーメッセージは `role="alert"` または `aria-live="polite"` を付与して スクリーンリーダー対応を行う
- 画像には `alt` テキストを必ず付与する（装飾目的の場合は `alt=""`）
- キーボード操作で全フォーム要素にアクセスできること

## 11. 禁止事項

| # | 禁止事項 | 理由 |
| --- | --- | --- |
| 1 | `any` 型の使用 | 型安全性が失われる |
| 2 | API URL のハードコード（`http://localhost:8080` 等） | 環境差異で動作しなくなる |
| 3 | `console.log` をコミットに含める | 本番ログ汚染・情報漏洩リスク |
| 4 | `fetch` を `try / catch` なしで呼び出す | ネットワーク断時にクラッシュする |
| 5 | グローバルスコープへの状態定義（モジュールトップレベルの `let`） | 複数インスタンス間で状態が共有される |
| 6 | `svelte-check` エラーを残したままコミット | 型エラーが本番に混入する |

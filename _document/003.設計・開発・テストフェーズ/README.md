## 2. Phase・Sprint・Ticket 分割基準

タスク分割の基準は [002-03.タスク推進計画](../002.要件定義フェーズ/002.プロジェクト計画/002-03.タスク推進計画.md) で定義する。

本ディレクトリの対象は **Musuhi Version 1.0.0（UC-01 新規プロジェクト開発）** のみとする。
Version 2.0.0 以降（UC-02〜UC-04）は [001-01.機能要件定義書](../002.要件定義フェーズ/001.要件定義/001-01.機能要件定義書.md) で管理する。

| 分割単位 | 基準 |
| --- | --- |
| Phase | 1つのユースケースが完成するまでを 1 Phase とする |
| Sprint | 1つのユースケースを完成させるのに必要な 1 機能を完成させるのを 1 Sprint とする |
| Ticket | 1 Sprint 内に必要な 1 小機能の完成までを 1 Ticket とする |

Phase 0 は「提案・要求仕様・要件定義」の固定フェーズであり、タスク構造は固定。Version 1.0.0（UC-01）向けテンプレート詳細は [001-03.フェーズ・スプリント・チケット設計規約](001.開発規約/001-03.フェーズ・スプリント・チケット設計規約.md) を参照する。

## 4. 運用ルール要約

- 1チケットは 1成果物・1担当・1完了条件に分解する
- 依存関係は `Depends on` で必ず明示する
- ガントチャート確認は Roadmap ビューを正とする
- タスク完了ごとに `新規プロジェクト/_document/000.進捗状況` へ成果物を出力し、出力先と生成ファイル名を記録する
- Step12 の設計成果物は `新規プロジェクト/_document/004.リリース・運用フェーズ` 配下に保存する
- Step13 のレトロスペクティブ結果は `新規プロジェクト/_document/003.設計・開発・テストフェーズ` 配下に保存する
- 各タスク完了時に生成・更新ドキュメントを commit / push する
- 各スプリント完了時にレビュー記録を残す

## 1. テスト方針

| 方針 | 内容 |
| --- | --- |
| 単体テスト | Service 層・Handler 層を対象にモックを使用してテストする |
| 内部結合テスト | `httptest.Server` を使いルーティング〜ミドルウェア〜Handler の連結をテストする |
| DB 実機テスト | SP1-1 スコープでは実施しない（Repository 層はモック代替） |
| CI | `go test ./...` で全テストを実行し、全 PASS を確認する |
| カバレッジ目標 | Service 層・Handler 層 80% 以上 |

## 5. 受け入れ条件（AC）との対応

Issue TK1-1-2 の受け入れ条件との対応を示す。

| 受け入れ条件 | 対応テスト | 確認方法 |
| --- | --- | --- |
| AC-001: 機能・構成要素が概要から抽出される | UT-SVC-P01 / ITb-001 | `data.features` / `data.components` が返ること |
| AC-002: プロジェクト名候補が生成される | UT-SVC-P02〜P04 / ITb-005 | `data.candidates` / `data.items` が返ること |
| AC-003: AI 神名候補に理由が付与される | UT-SVC-P03 / UT-HDL-P02 | `items[0].aiSuggested == true` かつ `reason` 非空 |
| AC-004: 初期ディレクトリが作成される | UT-SVC-P06 / ITb-011 | `directoryStatus == "success"` かつ `.keep` が存在する |
| AC-005: モデル運用モードを取得・更新できる | UT-HDL-P06〜P07 / ITb-009〜010 | `name-suggestion-profile` の GET/PUT が 200 で応答すること |
| 単体テストが pass する | UT-SVC-P01〜P09 / UT-HDL-P01〜P07 | `go test ./internal/...` |
| 内部結合テストが pass する | ITb-001〜013 | `go test ./test/...` |

## 6. テスト実行方法

```bash
# すべてのテストを実行
cd services/musuhi-api/src
go test ./...

# 単体テストのみ
go test ./internal/...

# 内部結合テストのみ
go test ./test/...

# カバレッジ付き
go test ./internal/... -coverprofile=cover.out
go tool cover -html=cover.out
```

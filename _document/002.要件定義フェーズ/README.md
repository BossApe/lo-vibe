### 2.1 Phase・Iteration・Checket 分割基準

Musuhi での新規プロジェクト開発において、タスク分割を行う際の基準を定義する。
詳細は [002-03.Phase-Iteration-Checket分割基準](002.プロジェクト計画/002-03.Phase-Iteration-Checket分割基準.md) を参照する。

| 分割単位 | 基準 |
| --- | --- |
| Phase | 1つのユースケースが完成するまでを 1 Phase とする |
| Iteration | 1つのユースケースを完成させるのに必要な 1 機能を完成させるのを 1 Iteration とする |
| Checket | 1 Iteration 内に必要な 1 小機能の完成までを 1 Checket とする |

Phase 0 は「提案・要求仕様・要件定義」の固定フェーズとして必ず存在し、タスク構造は固定である（[4. Phase 0 固定タスク](002.プロジェクト計画/002-03.Phase-Iteration-Checket分割基準.md#4-phase-0固定タスク) 参照）。

## 7. 用語定義・前提

- 本ディレクトリで記載する `Phase 0 / Phase 1 / Phase 2` は、Musuhi による開発実行フェーズ（MVP 開発フェーズ）を指す
- [001.提案・要求仕様フェーズ](../001.提案・要求仕様フェーズ/README.md) に記載した「フェーズ0（プロジェクト立ち上げ）」とは別概念として扱う
- 本フェーズ開始時点で、提案・要求仕様フェーズ成果物はユーザ承認済みかつ GitHub（または相当ツール）へ保存済みであることを前提とする
- ドキュメントレビュー時の要約提示、UI 上の承認操作、GitHub/Projects への登録処理は、要件定義フェーズで機能仕様へ落とし込んで管理する

## 1. 目的

本ドキュメントは、Musuhi におけるログの出力・保存・運用ルールを定義する。
障害調査、監査対応、品質改善を可能にしつつ、機密情報漏えいリスクを抑制することを目的とする。

## 2. 適用範囲

- `services/` 配下の全サービス（API、フロントエンド、バッチ、ワーカー）
- `_compose/` で起動するミドルウェア連携ログ
- CI/CD 実行ログ（ビルド、テスト、デプロイ）
- 将来追加する運用スクリプトの実行ログ

## 3. ログレベル定義

| レベル | 用途 | 例 |
| --- | --- | --- |
| `DEBUG` | 開発時の詳細追跡 | 入出力の要約、分岐判定、SQL 実行時間 |
| `INFO` | 正常系イベント記録 | リクエスト開始/終了、ジョブ開始/終了 |
| `WARN` | 継続可能な異常 | リトライ発生、外部 API 一時失敗 |
| `ERROR` | 処理失敗 | 例外、タイムアウト、DB 失敗 |
| `FATAL` | 継続不能 | 起動失敗、設定不整合での停止 |

- 本番環境の標準ログレベルは `INFO` 以上とする
- `DEBUG` は一時的な調査用途のみ有効化し、調査後に無効化する

## 4. ログ出力フォーマット

### 4.1 共通必須項目

| 項目 | 説明 | 必須 |
| --- | --- | --- |
| `timestamp` | ISO 8601 形式（UTC） | 必須 |
| `level` | ログレベル | 必須 |
| `service` | サービス名 | 必須 |
| `environment` | `local/stg/prod` | 必須 |
| `trace_id` | リクエスト横断 ID | 推奨 |
| `span_id` | 処理単位 ID | 推奨 |
| `event` | イベント名 | 必須 |
| `message` | 人間向け要約 | 必須 |
| `error_code` | 業務/システムエラーコード | 任意 |
| `duration_ms` | 処理時間（ミリ秒） | 任意 |

### 4.2 出力形式（JSON Lines）

- 1 行 1 JSON の JSON Lines 形式を標準とする
- 改行を含むメッセージは禁止し、必要時は配列または別項目へ分離する
- 時刻は必ず UTC で記録する
- 文字コードは UTF-8 とする

出力例:

```json
{"timestamp":"2026-05-02T01:30:45Z","level":"INFO","service":"musuhi-api","environment":"stg","trace_id":"c9c2...","event":"project.create.completed","message":"project created","duration_ms":128}
```

### 4.3 マスキング・出力禁止情報

以下の情報はログ出力を禁止し、必要時はマスクする。

- パスワード、API キー、トークン、秘密鍵
- 個人情報（氏名、メールアドレス、電話番号、住所、IP の生値）
- 認証ヘッダ、Cookie、セッション ID
- SQL 生クエリのパラメータ生値

マスキング例:

- メールアドレス: `a***@example.com`
- トークン: `****`（長さ情報も出力しない）
- IP: `192.168.*.*`

## 5. ログ出力タイミング

### 5.1 API リクエスト境界

- リクエスト受信時に `request.received` を `INFO` で出力する
- レスポンス返却時に `request.completed` を `INFO` で出力する
- 4xx は `WARN`、5xx は `ERROR` で出力する
- `trace_id` を必ず引き回し、関連ログを相互参照可能にする

### 5.2 業務イベント

- 主要な業務処理の開始・成功・失敗を出力する
- 状態遷移（例: `draft -> approved`）は遷移前後を記録する
- 監査対象操作（承認、削除、権限変更）は実行主体を記録する

### 5.3 外部連携・バッチ処理

- 外部 API 呼び出し前後を出力する（宛先、処理時間、結果）
- バッチはジョブ単位で開始・終了・件数サマリを出力する
- 連携失敗時はリトライ回数、最終結果を記録する

### 5.4 障害・リトライ

- 例外発生時は `ERROR` で原因と影響範囲を記録する
- リトライ時は `WARN` で `retry_count` を記録する
- サーキットブレーカー開閉、フォールバック発動は必ず記録する

## 6. ログ保存規約

### 6.1 保存先と責務

| 種別 | 保存先 | 主用途 |
| --- | --- | --- |
| アプリケーションログ | 標準出力（stdout/stderr） | コンテナ実行ログ収集 |
| 集約ログ | Loki（または同等基盤） | 横断検索・可視化 |
| 監査ログ | 改ざん耐性のある保管領域 | 監査証跡 |
| CI/CD ログ | CI システム保管領域 | 変更追跡 |

- アプリケーションはファイル直書きを原則禁止し、標準出力へ出力する
- 集約先でサービス名・環境・期間で検索可能であることを必須とする

### 6.2 保持期間

| ログ種別 | 最低保持期間 |
| --- | --- |
| アプリケーションログ | 90日 |
| 障害調査用詳細ログ | 30日 |
| 監査ログ | 1年 |
| セキュリティ関連ログ | 1年 |

- 法令・契約上の要求がある場合は、その期間を優先する

### 6.3 ローテーション・削除

- 容量上限と保持日数の両方でローテーションする
- 削除は自動化し、手動削除は緊急時のみとする
- 手動削除時は理由・実施者・対象期間を監査ログへ記録する

## 7. ログ運用ルール

### 7.1 監視・アラート

- `ERROR` 発生率、5xx 率、レイテンシ閾値超過を監視対象とする
- 重要アラートはオンコール通知し、一次対応 SLA を定義する
- 同一エラーの連続発生は集約し、通知ストームを抑制する

### 7.2 アクセス制御

- ログ閲覧権限は最小権限原則で付与する
- 本番ログのダウンロードは承認制とし、操作履歴を残す
- 開発環境へ本番ログを複製する場合は匿名化を必須とする

### 7.3 レビュー・監査

- 新規機能 PR では「必要ログが出力されるか」をレビュー項目に含める
- 月次でログ出力過不足、マスキング漏れ、保持設定を棚卸しする
- インシデント対応後にログ改善項目を追記し、再発防止へ反映する

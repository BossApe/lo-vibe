## 1. 目的

本ドキュメントは、Musuhi 開発における実装・運用上のセキュリティ規約を定義する。
非機能要件定義書（NFR-SEC）の「何を守るか」に対して、「どう実装するか」の開発基準を確立し、OWASP Top 10 への準拠と継続的なセキュリティ品質維持を目的とする。

## 2. 適用範囲

- `services/` 配下の全サービス（API、フロントエンド、バッチ）
- `_compose/` で管理するコンテナ構成・ミドルウェア設定
- CI/CD パイプラインおよびリポジトリ運用

## 3. 秘匿情報管理

### 3.1 禁止事項

| 禁止行為 | 理由 |
| --- | --- |
| API キー・パスワードをソースコードへ直接記述 | リポジトリ流出リスク |
| `.env` ファイルを Git へコミット | 秘匿情報の公開 |
| ログへの秘匿情報出力 | ログ参照者への情報漏えい |
| Docker イメージへの秘匿情報埋め込み | イメージ配布時の流出 |

### 3.2 秘匿情報の取り扱い方法

- 開発環境: `.env` ファイルで管理し、`.gitignore` に必ず追加する
- `.env.example` をリポジトリへコミットし、必須環境変数と説明を記載する（値は空またはダミー）
- 本番環境: Docker Secret または外部シークレット管理（Vault 等）を使用する
- CI 環境: Forgejo Actions Secrets / 相当ツールのシークレット機能を使用する
- コミット前に `git-secrets` または `gitleaks` によるスキャンを実施する

```bash
# gitleaks によるスキャン例（CI に組み込む）
gitleaks detect --source . --no-git
```

## 4. 認証・認可

### 4.1 認証方式

| 対象 | 方式 | 規約 |
| --- | --- | --- |
| 管理系 API エンドポイント | JWT | アルゴリズムは `RS256` を使用。有効期限は最大 1 時間 |
| 外部連携（Forgejo等） | OAuth2 / PAT | スコープは最小権限に限定 |
| 内部サービス間 | mTLS または共有シークレット | Docker ネットワーク内に閉じる場合も認証を省略しない |

- JWT 署名鍵はハードコードせず、環境変数またはシークレット管理から取得する
- トークンは HTTPS 通信のみで送受信し、Cookie 使用時は `HttpOnly` + `Secure` フラグを必須とする
- リフレッシュトークンはローテーション方式を採用し、流出時に無効化できる仕組みを設ける

### 4.2 認可制御

- 役割（PO / 開発者 / 閲覧者）ごとに許可操作を定義し、ミドルウェアで一元制御する
- エンドポイント単位で必要な役割をコード内に明示する
- 認可チェックはリクエスト受信時に必ず実施する（クライアント側の制御のみに依存しない）

## 5. 入力バリデーション・出力エスケープ

### 5.1 入力バリデーション

- すべての API 入力はサーバサイドでバリデーションを実施する（クライアント側バリデーションは補助手段）
- バリデーション規則は機能要件定義書の入力項目定義に従う
- 想定外の型・長さ・フォーマットは 400 エラーで拒否し、詳細なエラー情報を内部ログに記録する（レスポンスには最小限の情報のみ返す）

| 対象 | バリデーション規則例 |
| --- | --- |
| 文字列 | 最大長チェック、制御文字・NULL バイト除去 |
| UUID | RFC 4122 形式チェック |
| 日付 | ISO 8601 形式チェック、未来/過去の範囲チェック |
| Enum | 定義済み値のみ許可 |
| ファイルアップロード | MIME タイプ・拡張子・サイズ制限を実施 |

### 5.2 出力エスケープ

- HTML レンダリング箇所では XSS を防ぐためエスケープ処理を実施する
- Markdown レンダリングは信頼済みライブラリを使用し、`<script>` タグ等の埋め込みを無効化する
- JSON レスポンスに HTML 特殊文字が含まれる場合はエスケープする

### 5.3 SQL インジェクション対策

- すべての DB クエリはプレースホルダ（プリペアドステートメント）を使用する
- ORM 使用時も生 SQL への文字列連結は禁止する
- ユーザ入力を ORDER BY / テーブル名等に動的に使用する場合は許可リスト方式を採用する

## 6. 通信暗号化

- 外部公開エンドポイントは Traefik による TLS 終端（HTTPS）を必須とする（NFR-SEC-005）
- HTTP から HTTPS へのリダイレクトを強制する
- TLS バージョンは 1.2 以上を使用し、1.0 / 1.1 は無効化する
- 内部サービス間通信はプライベートネットワーク内に閉じ、外部公開しない
- 機密性の高いデータ（パスワード等）は保存前に bcrypt または Argon2 でハッシュ化する

## 7. 依存ライブラリ管理

- サードパーティライブラリは既知脆弱性がないことを CI で確認する（NFR-SEC-007）
- 使用ツール:
  - Go: `govulncheck`
  - コンテナイメージ: `Trivy`
  - フロントエンド: `npm audit`
- 高深刻度（CVSS >= 7.0）の脆弱性は発見後 7 日以内に対応する
- ライブラリのバージョンは `go.mod` / `package.json` に固定し、意図しないアップデートを防ぐ
- 本番環境で不要な開発用依存は `devDependencies` / ビルドステージに分離する

## 8. コンテナ・インフラセキュリティ

### 8.1 コンテナ設計

| 規約 | 内容 |
| --- | --- |
| 非 root 実行 | コンテナプロセスは `USER` 命令で非 root ユーザとして実行する |
| イメージの最小化 | distroless または alpine ベースを優先し、不要なツールを含めない |
| 読み取り専用ファイルシステム | 書き込みが不要なコンテナは `read_only: true` を設定する |
| ヘルスチェック | すべてのサービスコンテナに `HEALTHCHECK` を定義する |
| イメージスキャン | CI で Trivy によるイメージ脆弱性スキャンを実施する |

### 8.2 ネットワーク制御

- Docker ネットワークは目的別に分離し、サービス間の不要な通信を遮断する
- 外部公開ポートは API ゲートウェイ（Kong）のみとし、各サービスポートを直接公開しない
- 環境変数でホスト・ポートを可変にし、本番環境での直接アクセスを防ぐ

## 9. ログ・監査

- 認証失敗・認可拒否・不正入力は `WARN` または `ERROR` レベルでログ出力する
- ログに含めてはならない情報: パスワード、トークン、個人情報（005-04.ログ規約 4.3 参照）
- 管理操作（設定変更・ユーザ権限変更等）は監査ログとして記録する
- ログの改ざんを防ぐため、ログ書き込み権限はサービスアカウントに限定する

## 10. セキュリティレビュー観点

コードレビュー・PR マージ時に以下を確認する。

- [ ] 秘匿情報がコード・コミットに含まれていないか
- [ ] すべての API 入力にバリデーションが実装されているか
- [ ] SQL クエリにプレースホルダを使用しているか
- [ ] 認証・認可チェックが適切に実装されているか
- [ ] 新規依存ライブラリに既知脆弱性がないか（`govulncheck` / `npm audit` 確認）
- [ ] エラーレスポンスに内部情報（スタックトレース等）が含まれていないか
- [ ] コンテナが非 root ユーザで実行されているか
- [ ] ログ出力に秘匿情報・個人情報が含まれていないか

> **認証/権限/データ永続化/外部公開経路に関わる変更は手動レビューを必須とする**（FR-011 AC-011 参照）

## 1. 目的

本ドキュメントは、Musuhi 開発における統一されたコーディング規約を定義する。
コードの可読性・保守性・セキュリティを確保し、レビュー品質を高める。

## 2. 共通規約

- コードはすべて英語で記述する（コメントは日本語可）
- 1 関数・1 メソッドの責務は単一とする（単一責任原則）
- マジックナンバーは定数として定義する
- 機密情報（パスワード・API キー・シークレット）をコードに直接記述しない
- `.env` ファイルにも平文の機密情報を保存せず、Docker Secret または環境変数管理ツールを使用する
- 依存関係の脆弱性チェックを CI で必須とする

## 3. Go（バックエンド API）規約

### 3.1 スタイル・フォーマット

- `gofmt` / `goimports` でフォーマットする（CI で強制）
- `golangci-lint` で静的解析を実施する（CI で強制）
- 1行の長さは 120文字以内を目安とする

### 3.2 命名規則

| 種別 | 規則 | 例 |
| --- | --- | --- |
| パッケージ名 | 小文字・単数形 | `project`, `session` |
| ファイル名 | snake_case | `project_handler.go` |
| 型名・インターフェース名 | PascalCase | `ProjectService`, `Repository` |
| 関数・メソッド名 | PascalCase（公開）/ camelCase（非公開） | `CreateProject`, `validate` |
| 変数名 | camelCase | `projectID`, `createdAt` |
| 定数名 | PascalCase または UPPER_SNAKE_CASE | `MaxNameLength`, `MAX_RETRY` |
| エラー変数名 | `Err` プレフィックス | `ErrNotFound`, `ErrConflict` |

### 3.3 パッケージ構成

```
musuhi-api/src/
├── main.go
├── internal/
│   ├── handler/      # HTTP ハンドラ
│   ├── service/      # ビジネスロジック
│   ├── repository/   # データアクセス
│   ├── model/        # ドメインモデル
│   └── middleware/   # 認証・ロギング等
├── pkg/
│   ├── logger/       # ロガー
│   └── validator/    # バリデーター
└── config/           # 設定読み込み
```

- `internal/` 配下は外部パッケージから参照不可とする
- `handler -> service -> repository` の依存方向を維持する（逆方向禁止）

### 3.4 エラーハンドリング

```go
// ○ エラーをラップして文脈を付与する
if err != nil {
    return fmt.Errorf("createProject: %w", err)
}

// × エラーを無視しない
result, _ := someFunc()  // NG
```

- エラーは必ず処理するか、呼び出し元へ返す
- `errors.Is` / `errors.As` でエラー種別を判定する
- HTTP ハンドラでは共通エラーレスポンス形式で返す

### 3.5 セキュリティ

- SQL は必ずプレースホルダー（パラメータバインディング）を使用する（SQL インジェクション防止）

```go
// ○
db.QueryRow("SELECT * FROM projects WHERE id = $1", id)

// × 文字列結合禁止
db.QueryRow("SELECT * FROM projects WHERE id = " + id)
```

- HTML をレスポンスに含める場合は必ずエスケープする
- JWT シークレットは環境変数から取得する
- 入力値は handler 層でバリデーションし、service 層では検証済みとして扱う

### 3.6 テスト

- テスト関数名は `Test{対象関数名}_{シナリオ}` とする
- テーブル駆動テストを積極的に使用する
- 外部依存（DB・外部 API）はインターフェースを介してモックする
- `testify/assert` を標準利用する

```go
func TestCreateProject_Success(t *testing.T) { ... }
func TestCreateProject_ValidationError(t *testing.T) { ... }
```

### 3.7 テストコードの保存場所

- 単体テスト（Unit Test）は、対象実装と同じレイヤ/パッケージ配下に `*_test.go` として配置する
  - 例: `internal/handler/*_test.go`, `internal/service/*_test.go`, `internal/repository/*_test.go`
- 内部結合テスト（Integration Test）は、`test/integration/` 配下に `*_test.go` として配置する
  - 例: `test/integration/system_overview_test.go`, `test/integration/project_test.go`
- 外部結合テスト（External Integration Test）は、`test/external/` 配下に `*_test.go` として配置する
  - 例: `test/external/github_api_test.go`
- 受け入れテスト（Acceptance Test）は、`test/acceptance/` 配下に `*_test.go` として配置する
  - 例: `test/acceptance/ticket_flow_test.go`
- E2E テストを追加する場合は、`test/e2e/` 配下に配置する
- テスト種別の混在を禁止する
  - 単体テストファイルに内部結合テストケースを追加しない
  - 内部結合テストファイルに単体テスト向けのモック前提ケースを追加しない
- 新規テスト追加時は、上記ディレクトリ規約に従い、既存ファイルへ追記するか同階層に新規作成する
- 各テスト実施時に Coverage Report を markdown 形式で出力し、テスト結果保存ディレクトリに `coverage_report.md` として保存する
  - 形式: `go test -cover` の出力をまとめた markdown（例: `Unit Test Coverage: 85.5%`, パッケージごとの カバレッジ率）
  - 保存先: 
    - 単体テスト: `新規プロジェクト/_document/003.設計・開発・テストフェーズ/004.テスト結果/001.単体テスト/coverage_report.md`
    - 内部結合テスト: `新規プロジェクト/_document/003.設計・開発・テストフェーズ/004.テスト結果/002.内部結合テスト/coverage_report.md`
    - 外部結合テスト: `新規プロジェクト/_document/003.設計・開発・テストフェーズ/004.テスト結果/003.外部結合テスト/coverage_report.md`
    - 受け入れテスト: `新規プロジェクト/_document/003.設計・開発・テストフェーズ/004.テスト結果/004.受け入れテスト/coverage_report.md`

## 4. TypeScript（フロントエンド）規約

### 4.1 スタイル・フォーマット

- `Prettier` でフォーマットする（CI で強制）
- `ESLint` で静的解析を実施する（CI で強制）
- `strict: true` を TypeScript コンパイラオプションで有効にする

### 4.2 命名規則

| 種別 | 規則 | 例 |
| --- | --- | --- |
| ファイル名（コンポーネント） | PascalCase | `ProjectList.svelte` |
| ファイル名（その他） | camelCase | `apiClient.ts` |
| 型・インターフェース名 | PascalCase | `Project`, `ApiResponse` |
| 関数名 | camelCase | `createProject`, `fetchProjects` |
| 変数名 | camelCase | `projectId`, `isLoading` |
| 定数名 | UPPER_SNAKE_CASE | `MAX_NAME_LENGTH` |
| CSS クラス名 | kebab-case | `project-list-item` |

### 4.3 コンポーネント設計

- コンポーネントは単一責務とする
- props は型を明示する
- API 通信は `src/lib/api/` 配下のクライアントモジュールに集約する

### 4.4 セキュリティ

- `innerHTML` の直接操作を禁止する（XSS 防止）
- ユーザー入力をそのまま DOM に挿入しない
- API キー・シークレットをフロントエンドコードに含めない
- SvelteKit の CSP ヘッダを設定する

## 5. SQL・データベース規約

- テーブル名・カラム名は snake_case とする
- 主キーは UUID（`gen_random_uuid()`）を使用する
- 全テーブルに `created_at`, `updated_at` を設ける
- 削除は論理削除（`deleted_at`）を基本とする
- インデックスは外部キー・検索頻度の高いカラムに設ける
- マイグレーションは `goose` を使用し、`up` / `down` を必ず実装する
- マイグレーションファイル名: `{連番}_{説明}.sql`（例: `001_create_projects.sql`）

## 6. Docker・インフラ規約

- Dockerfile は可能な限りマルチステージビルドを使用する
- 本番イメージは `distroless` または最小 Alpine を使用する
- コンテナは root 以外のユーザーで実行する
- 環境変数で動作を制御し、設定のハードコーディングを禁止する
- ヘルスチェックを全コンテナに定義する
- コンテナイメージはタグでバージョンを固定する（`latest` 禁止）

## 7. CI での強制

以下は PR マージ時に CI で自動チェックし、失敗時はマージをブロックする。

| チェック内容 | ツール |
| --- | --- |
| Go フォーマット | `gofmt` |
| Go 静的解析 | `golangci-lint` |
| Go 脆弱性チェック | `govulncheck` |
| TS フォーマット | `prettier` |
| TS 静的解析 | `eslint` |
| コンテナ脆弱性スキャン | `trivy` |
| テストカバレッジ | `go test -cover` / `vitest --coverage` |

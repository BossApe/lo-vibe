## 1. 目的

本ドキュメントは、Musuhi プロジェクトにおけるドキュメント作成・管理の規約を定義する。
ドキュメントの品質と一貫性を保ち、チーム全員が同じ基準で作成・更新できるようにする。

## 2. ドキュメント管理方針

- ドキュメントは `_document/` 配下でフェーズ別に管理する
- 変更は Git で管理し、master マージ前に必ず更新する
- 実装との同期を毎 Sprint で確認する
- 用語は [003-04.用語集・前提定義](../../001.提案・要求仕様フェーズ/003.要求仕様共通/003-04.用語集・前提定義.md) に従う
- 新規用語は用語集に先に追加してから使用する
- `tools/save_promptinglog/` 配下（AI セッションのプロンプトログ）は業務ドキュメントではなく作業記録として扱い、リンク整合チェック・命名規約・レビューチェックリストの品質管理対象外とする

## 3. ディレクトリ・ファイル命名規則

### ディレクトリ命名

```
{3桁連番}.{ディレクトリ名}

例:
  001.提案・要求仕様フェーズ/
  002.要件定義フェーズ/
  002.プロジェクト計画/
```

### ファイル命名

```
{3桁連番}-{2桁連番}.{ファイル名}.md

例:
  002-01.プロジェクト計画書.md
  001-01.機能要件定義書.md
  005-03.Git運用規約.md
```

- ファイル名に使用できる文字: 英数字・日本語・ハイフン・アンダースコア・ドット
- スペースは使用しない
- 大文字・小文字は統一する（英数字は小文字を原則とする）

## 4. Markdown 記述規則

### 4.1 ファイル構造テンプレート

すべてのドキュメントファイルは以下の構造に従う。

```markdown
# {ドキュメントタイトル}

{前後ナビゲーション（前: / 一覧: / 次:）}

<details>
<summary>目次（クリックで展開）</summary>

- [1. セクション名](#1-セクション名)
  - [1.1 サブセクション](#11-サブセクション)

</details>

## 1. セクション名

本文

### 1.1 サブセクション

本文

## N. 更新履歴

| 日付 | 版 | 変更内容 | 作成者 |
| --- | --- | --- | --- |
| YYYY-MM-DD | 0.1 | 初版作成 | 作成者名 |
```

### 4.2 見出し規則

| レベル | 用途 |
| --- | --- |
| `#` (H1) | ドキュメントタイトル（1ファイルに1つ） |
| `##` (H2) | 主要セクション（番号付き: `## 1. 目的`） |
| `###` (H3) | サブセクション（番号付き: `### 1.1 ...`） |
| `####` (H4) | 詳細項目（必要な場合のみ） |

- セクション番号は `1.` から始め、連番とする
- 最後のセクションは必ず「更新履歴」とする

### 4.3 表記統一

| 項目 | 規則 | 例 |
| --- | --- | --- |
| 半角・全角スペース | 英数字と日本語の間にスペースなし（文内）| `Musuhi の主要機能` → `Musuhi の主要機能`（Markdown レンダリング時） |
| 句読点 | 「。」「、」を使用（箇条書き末尾は省略可） | |
| 英語固有名詞 | 正式表記に従う | `GitHub`、`Docker`、`PostgreSQL` |
| ID 表記 | ハイフン区切り・大文字プレフィックス | `FR-001`、`AC-001`、`NFR-PERF-001` |
| 日付形式 | ISO 8601 | `2026-05-01` |

### 4.4 図・ダイアグラム

- フロー図・シーケンス図・ER 図は Mermaid を使用する
- Mermaid コードブロックは ` ```mermaid ` で囲む
- 複雑な図は Markdown 内に配置し、外部ファイル参照は `img/` サブディレクトリに保管する

**Mermaid 使用推奨パターン:**

| 図の種類 | Mermaid 記法 | 用途 |
| --- | --- | --- |
| フローチャート | `flowchart TD/LR` | 処理フロー・アーキテクチャ |
| シーケンス図 | `sequenceDiagram` | コンポーネント間連携 |
| ER 図 | `erDiagram` | データ定義 |
| ガントチャート | `gantt` | スケジュール |

### 4.5 リンク

- ドキュメント間のリンクは相対パスで記述する
- リンクテキストはファイル名またはドキュメントタイトルを使用する
- 外部 URL は正式 URL を使用する
- リンク切れを定期的にチェックする（CI または手動）

```markdown
# ○ 相対パスリンク
[001-01.機能要件定義書](../001.要件定義/001-01.機能要件定義書.md)

# × 絶対パスリンク（環境依存のため禁止）
[機能要件定義書](/absolute/path/003-01.md)
```

## 5. フェーズ別ドキュメント構成

| フェーズ | ディレクトリ | 主要ドキュメント |
| --- | --- | --- |
| 001.提案・要求仕様フェーズ | `001.提案・要求仕様フェーズ/` | アプリケーション提案・インフラ案・要求仕様共通 |
| 002.要件定義フェーズ | `002.要件定義フェーズ/` | プロジェクト計画・品質管理・要件定義・テスト計画・規約 |
| 003.設計フェーズ（予定） | `003.設計フェーズ/` | アーキテクチャ設計・詳細設計・DB 設計 |

各フェーズのディレクトリには `README.md`（インデックス）を必ず配置する。

## 6. 更新ルール

- ドキュメントは master マージ前に更新する（ドキュメント反映が先）
- 更新履歴は必ず記録する（日付・版・変更内容・作成者）
- 版番号は `0.1` から始め、0.1 ずつインクリメントする
- 大幅な改訂時は 1.0 にメジャーバンプする
- 変更が他ドキュメントに影響する場合は同時更新する
- `Musuhi/tools/` はプロジェクト推進に必要な各種バッチ・補助スクリプトの保管場所として扱う
- `tools` 配下へ資材を追加する際は、同一ディレクトリに `README.md` を作成し、`概要` `目的` `利用方法` `運用手順` を必須記載する

## 7. レビュー観点

ドキュメントレビュー時は以下を確認する。

- [ ] ファイル構造テンプレートに準拠しているか
- [ ] 見出しレベルと番号が正しいか
- [ ] 更新履歴が記録されているか
- [ ] リンクが有効か（リンク切れがないか）
- [ ] 用語集に沿った用語を使用しているか
- [ ] ID（FR/AC/NFR）の表記が統一されているか
- [ ] Mermaid 図が正しくレンダリングされるか
- [ ] 関連ドキュメントとの整合性が取れているか

> **対象外**: `tools/save_promptinglog/` 配下のファイルは上記チェック項目の適用対象外とする。

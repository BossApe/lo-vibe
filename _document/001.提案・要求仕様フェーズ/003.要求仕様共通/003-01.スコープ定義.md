## 1. 目的

本ドキュメントは、Musuhi の提案・要求仕様フェーズにおける開発対象範囲を明確化する。

## 2. 対象リリース

- 対象: Phase 0 (MVP)
- 対象イテレーション: Iteration 1 から Iteration 3
- 1 イテレーションの定義: 計画 -> 実装 -> テスト -> レビュー -> ドキュメント反映 -> リリース (master マージ)
- 対象プロダクト: Musuhi

## 3. In Scope

以下は本リリースで実施する。

| ID | 項目 | 概要 | 優先度 | 備考 |
| --- | --- | --- | --- | --- |
| IN-001 | プロジェクト管理の基本機能 | プロジェクト作成、更新、一覧表示 | High | FR-001 |
| IN-002 | プロンプトログ保存・表示 | セッション単位で保存し再表示可能 | High | FR-002 |
| IN-003 | Markdown ドキュメント管理 | 作成、編集、履歴参照 | High | FR-003 |
| IN-004 | aider 基本連携 | 実装タスク分割後に指示連携できる | Medium | FR-004 |
| IN-005 | 進捗の可視化 | マイルストーン単位で進捗確認可能 | Medium | FR-005 |

## 4. Out of Scope

以下は本リリースでは実施しない。

| ID | 項目 | 理由 | 将来対応 |
| --- | --- | --- | --- |
| OUT-001 | マルチテナント対応 | 初期は少人数運用のため不要 | Phase 2 |
| OUT-002 | 高度な RBAC と監査統合 | MVP では認証最小構成を優先 | Phase 2 |
| OUT-003 | 外部ツール広域連携 (Slack 等) | 外部連携は段階導入方針 | Phase 2 |
| OUT-004 | 自動レポート高度分析 | まず手動レビューを前提 | Phase 1 以降 |
| OUT-005 | レガシーシステム改修支援 | MVP では新規開発サポートを優先するため | Phase 1 |

## 5. 前提条件

- Linux サーバ + Docker Compose 環境を利用可能であること
- 既存ドキュメント構成に従うこと
- AI 活用を前提とした開発体制であること
- 002.インフラ構成案で定義した最小構成 (Traefik, Open WebUI, Ollama, aider, Forgejo / Gitea / GitHub) が動作すること
- Hugging Face の承認済みモデルリストと Ollama 取り込み手順が定義済みであること

## 6. 制約条件

- オープンソース中心で構成する
- 初期は少人数運用を前提とする
- コストと運用負荷を抑えた段階導入を行う
- 開発サイクル短縮目標 (従来比 40% 削減) を評価可能な形でログを残す

## 7. 依存関係

| ID | 依存対象 | 内容 | 影響 |

## 8. 変更管理

- スコープ変更は Issue 起票後に承認する
- In/Out Scope の変更時は本書を更新する
- 更新履歴を残す

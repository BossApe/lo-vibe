## 1. 目的

本ドキュメントは、Musuhi の機能要件を ID 管理し、設計・実装・テストへトレース可能にする。

## 2. 要件レベル定義

- Must: MVP で必須
- Should: 可能であれば実施
- Could: 将来拡張

対象スコープは 003-01.スコープ定義の In Scope を基準とする。
受け入れ判定は各要件の目標イテレーション終端で実施する。

### 4.2 要件詳細 (初版)

- 要件ID: FR-001
- 機能名: プロジェクト作成・一覧
- 背景/目的: 複数プロジェクトを並行管理するための基礎機能
- 入力: プロジェクト名、説明、開始日
- 出力: プロジェクトID、作成日時、初期状態
- 正常系フロー: 入力 -> 検証 -> 保存 -> 一覧表示
- 例外系フロー: 必須入力不足時にエラー表示
- 依存要件: なし
- 非機能要件との関係: 1秒以内の一覧表示
- 受け入れ基準ID: AC-001

- 要件ID: FR-002
- 機能名: プロンプトログ保存・再表示
- 背景/目的: AI 対話の再利用性と監査性を確保する
- 入力: セッションID、発話、応答、タイムスタンプ
- 出力: セッション単位の保存データ
- 正常系フロー: 発話/応答受信 -> 保存 -> 一覧 -> 再表示
- 例外系フロー: 保存失敗時にリトライまたは通知
- 依存要件: FR-001
- 非機能要件との関係: 永続保存、監査ログ
- 受け入れ基準ID: AC-002

- 要件ID: FR-003
- 機能名: Markdown 文書管理
- 背景/目的: 要求仕様から設計までの成果物を一元管理する
- 入力: 文書タイトル、本文、更新コメント
- 出力: 文書本体、履歴情報
- 正常系フロー: 作成/更新 -> 履歴記録 -> 表示
- 例外系フロー: 競合時の差分提示
- 依存要件: FR-001
- 非機能要件との関係: 履歴整合性、可用性
- 受け入れ基準ID: AC-003

- 要件ID: FR-008
- 機能名: レガシーシステム改修支援
- 背景/目的: 既存システムの修正・改善を AI で効率化する
- 入力: 対象コード・既存ドキュメント、分析視点指定
- 出力: 不足ドキュメント一覧、改善提案リスト、問題点一覧と修正案、テストコード、テスト仕様書、テスト結果レポート、Coverage レポート、IaC テンプレート
- 正常系フロー: コード/ドキュメント追加 -> AI 分析 -> 問題点・改善案出力 -> テスト整備・自動化 -> IaC 化 -> レビュー -> 修正対応
- 例外系フロー: 対象コンテキスト晲なしの場合は入力要求エラーを表示
- 依存要件: FR-002, FR-003, FR-004
- 非機能要件との関係: プロンプトログ永続保存、ドキュメント自動生成
- 受け入れ基準ID: AC-008

## 1. 目的

本ドキュメントは、Musuhi プロジェクトにおける主要用語と前提条件を統一し、認識ズレを防止する。

## 2. 運用ルール

- 新規用語は本書へ追記してから利用する
- 定義変更時は影響ドキュメントを同時更新する
- 略語は初出時に正式名称を併記する

## 3. 用語集

| 用語 | 定義 | 補足 |
| --- | --- | --- |
| Musuhi | ローカルバイブコーディング環境を実現する統合開発基盤 | プロジェクト名 |
| バイブコーディング | AI と対話しながら要件・設計・実装・テストを進める開発手法 | 本プロジェクトの中心手法 |
| Phase | 複数スプリントを束ねる開発段階 | Phase 0, 1, 2 |
| MVP | 最小機能で価値検証を行う初期リリース範囲 | Phase 0 に相当 |
| Sprint | 計画 -> 実装 -> テスト -> レビュー -> ドキュメント反映 -> master マージの1サイクル | 進行管理の最小単位 |
| FR | 機能ID (Functional Requirement) | 003-02 を参照 |
| AC | 受け入れ基準ID (Acceptance Criteria) | 003-03 を参照 |
| In Scope | 現在のリリースで実施対象とする範囲 | 003-01 を参照 |
| Out of Scope | 現在リリースでは実施しない範囲 | 003-01 を参照 |
| OSS | オープンソースソフトウェア | 採用方針の前提 |
| tools ディレクトリ | プロジェクト推進に必要なバッチ・補助スクリプトの保管場所 | `Musuhi/tools/` を指す |
| master マージ | 受け入れ完了後に master へ統合する行為 | スプリント完了条件 |
| ドキュメント反映 | 実装内容に合わせて仕様文書を更新する作業 | master マージ前に実施 |

## 4. 前提定義

### 4.1 開発体制の前提

- 開発・テストは Copilot 主体で実施する
- 人間は方針決定、レビュー、承認を主担当とする
- 仕様と実装の同期は毎スプリントで行う

### 4.2 技術前提

- Linux サーバ上で Docker Compose を利用する
- 主要構成要素は OSS を優先採用する
- compose 定義とサービス資材はディレクトリを分離して管理する
- `Musuhi/tools/` は各種バッチ・補助スクリプトの保管場所として運用する
- `tools` 配下へ追加する資材には、同ディレクトリの `README.md` で概要・目的・利用方法・運用手順を必須記載する

### 4.3 プロセス前提

- 受け入れ基準を満たした要件のみリリース対象とする
- 受け入れ判定はスプリント終端で行う
- 変更要求は変更管理ルールに従って処理する
